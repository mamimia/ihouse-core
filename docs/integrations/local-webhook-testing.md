# Local Webhook Testing

## Purpose

This note explains how to test webhook-based integrations during local development before the system is deployed on a real public HTTPS domain.

This is especially relevant for providers such as:
- LINE
- Telegram webhooks if applicable
- future WhatsApp integrations
- any third-party platform that must call our backend from the public internet

## Core limitation

A local development backend is not directly reachable by third-party webhook providers.

These addresses are not enough for real webhook delivery:
- localhost
- 127.0.0.1
- 192.168.x.x
- plain HTTP endpoints

Most providers require:
- a public URL
- valid HTTPS

This means:
- local code can be built and prepared
- but real inbound webhook proof requires a public HTTPS endpoint

## Development solution

Use a temporary HTTPS tunnel.

Recommended option:
- ngrok

Other temporary tunnels may work, but for webhook testing we prefer a provider that does not inject blocking challenge pages and is stable for machine-to-machine callbacks.

## ngrok local setup

### 1. Install ngrok

On macOS:
```bash
brew install ngrok
```

### 2. Authenticate

```bash
ngrok config add-authtoken YOUR_NGROK_AUTH_TOKEN
```

### 3. Start tunnel

Make sure the backend is running on port 8000, then start the tunnel:
```bash
ngrok http 8000
```

### 4. Provide URL to integration provider

Copy the HTTPS forwarding URL generated by ngrok (e.g., `https://your-ngrok-url.ngrok-free.app`) and paste it into the webhook settings of your integration provider along with the exact webhook route (e.g., `/line/webhook`).

### 5. Verify the delivery

Send a real test callback from the platform (e.g., send a text to the LINE bot) and monitor the backend logs for the request.
