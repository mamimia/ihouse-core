# iHouse Core — Canonical Event Architecture

## Status
Authoritative

Current:
Phase 23 — External Event Semantics Hardening (Open)

Last closed:

## Purpose
Define the single allowed external business event surface.
Define the single canonical persistence contract.
Define the single canonical operational apply path.

## Event Classes

### Domain Event (External Business Event)
Domain events represent business facts and business intent.
They are part of the external contract of the system.

Rules:
Domain events may be emitted by external producers.
Domain events are accepted only through the canonical apply gate.
Domain events are stored in the canonical event store.

### Projection Event (Internal)
Projection events are internal implementation artifacts.
They represent materialized state changes in projections.

Rules:
Projection events are generated only by the canonical database apply gate.
Projection events are never accepted from external producers.
Projection events exist only to materialize read models deterministically.

## Canonical Business Event Types (External)
BOOKING_CREATED
BOOKING_UPDATED
BOOKING_CANCELED
BOOKING_CHECKED_IN
BOOKING_CHECKED_OUT
BOOKING_SYNC_ERROR
AVAILABILITY_UPDATED
RATE_UPDATED

No additional external event types are allowed without registry update and validation.

## External Event Sources
Any external producer must emit canonical business events only.

External sources include:
- OTA channels
- Channel managers
- Internal admin tools
- User self-booking
- Manual bookings

All external sources must flow through the canonical Supabase apply gate.

External payloads must pass through the canonical ingestion adapter layer before reaching the apply gate.

## External Ingestion Adapter Layer (Phase 22)

External producers never interact directly with the canonical event store.

External payload flow:

External System  
→ Channel Adapter  
→ Payload Normalization  
→ Canonical Validation  
→ Canonical Envelope  
→ apply_envelope RPC  
→ event_log

The adapter layer functions as an anti-corruption boundary between external systems and the deterministic event kernel.

Responsibilities of the adapter layer:

- Normalize external payloads into canonical domain events
- Validate canonical schema and event rules
- Assign deterministic idempotency keys
- Produce canonical envelopes for apply_envelope

External payload formats must never leak into the canonical domain model.

## Internal Events
Internal events may exist as implementation detail.
They are never accepted from the public API.

STATE_UPSERT is a projection event used to materialize booking_state.
It is generated by the canonical apply gate and must not be sent by external producers.

## External Envelope Contract
An external envelope must contain:

type  
payload  
idempotency.request_id  

The system derives:

envelope_id from idempotency.request_id

## Supabase Authority
Supabase public.event_log is the single canonical event store.

All rebuild and replay derive exclusively from this table.

SQLite is forbidden as a production write path.

## Idempotency Gate
apply_envelope is the canonical atomic write gate.

It writes envelope_received once per envelope_id.

It prevents duplicate application by returning ALREADY_APPLIED.

Duplicate envelope replay must not:

insert additional events  
mutate booking_state

## State Mutation Authority
State mutation authority resides in the database apply gate.

The application layer is not allowed to synthesize internal state mutation events.

All booking_state changes must originate from the canonical apply path.

## Availability Canon (Phase 18)

Availability is evaluated at:

(tenant_id, property_id)

Canonical booking range is half-open:

[check_in, check_out)

Overlap predicate:

existing.check_in < new.check_out AND new.check_in < existing.check_out

Canonical active predicate for availability (legacy tolerant):

status IS DISTINCT FROM 'canceled'

Forward-only write rules:

On BOOKING_CREATED  
always write status = 'active'

On BOOKING_CANCELED  
set status = 'canceled'
