# iHouse Core — Canonical Event Architecture

## Status
Authoritative

Last closed:
Phase 18 — Legacy-Tolerant Availability Canon + DB Invariants (Closed)

## Purpose
Define the single allowed external business event surface.
Define the single canonical persistence contract.
Define the single canonical operational apply path.

## Canonical Business Event Types (External)
BOOKING_CREATED
BOOKING_UPDATED
BOOKING_CANCELED
BOOKING_CHECKED_IN
BOOKING_CHECKED_OUT
BOOKING_SYNC_ERROR
AVAILABILITY_UPDATED
RATE_UPDATED

No additional external event types are allowed without registry update and validation.

## External Event Sources
Any external producer must emit canonical business events only.
This includes channel managers, internal admin tools, user self-booking, and manual bookings.
All external sources must flow through the canonical Supabase apply gate.

## Internal Events
Internal events may exist as implementation detail.
They are never accepted from the public API.

STATE_UPSERT is an internal event used to materialize booking_state.
It is generated by the canonical apply gate and must not be sent by external producers.

## External Envelope Contract
An external envelope must contain:
type
payload
idempotency.request_id

The system derives:
envelope_id from idempotency.request_id

## Supabase Authority
Supabase public.event_log is the single canonical event store.
All rebuild and replay derive exclusively from this table.
SQLite is forbidden as a production write path.

## Idempotency Gate
apply_envelope is the canonical atomic write gate.
It writes envelope_received once per envelope_id.
It prevents duplicate application by returning ALREADY_APPLIED.

Duplicate envelope replay must not:
insert additional events
mutate booking_state

## State Mutation Authority
State mutation authority resides in the database apply gate.
The application layer is not allowed to synthesize internal state mutation events.
All booking_state changes must originate from the canonical apply path.

## Availability Canon (Phase 18)
Availability is evaluated at (tenant_id, property_id).

Canonical booking range is half-open:
[check_in, check_out)

Overlap predicate:
existing.check_in < new.check_out AND new.check_in < existing.check_out

Canonical active predicate for availability (Option B, legacy tolerant):
status IS DISTINCT FROM 'canceled'

This intentionally treats NULL as active for legacy rows (forward-only, no backfill).

Forward-only write rules:
On BOOKING_CREATED: always write status = 'active' for new rows.
On BOOKING_CANCELED: set status = 'canceled' and bump version under row lock, updating last_event_id and last_envelope_id.

## Business Identity Canon (Phase 17C)
Stable business identity is:
tenant_id + source + reservation_ref + property_id

This is the canonical dedup key for booking creation.
