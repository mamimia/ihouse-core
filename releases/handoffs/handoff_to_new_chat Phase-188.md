## Next Phase

**Phase 189 — Booking Mutation Audit Events**

The owner statement is now a proper document. Before building the Manager UI (Phase 190), we need actor attribution on every mutation endpoint — who canceled a booking, who amended it, who changed a task status.

### Objective

Add an audit record to every mutation endpoint:
- `POST /bookings/{id}/cancel`
- `POST /bookings/{id}/amend`
- `POST /tasks/{id}/transition`

Each record: `booking_id`, `actor_id` (from JWT `sub`), `actor_role` (from `permissions` table), `action` ("CANCEL", "AMEND", "TASK_TRANSITION"), `payload_snapshot` (what changed), `occurred_at`.

This can go into an `audit_events` table in Supabase (append-only, RLS service_role only).

### Why this before Manager UI (Phase 190)?

The Manager UI needs to display "who did what" on bookings. If we build the UI first, it has nothing to show. The audit trail is the data source for Manager Dashboard activity feed.

### Implementation hints

1. Check if `audit_events` table already exists in Supabase  
2. If not: migration `20260310_phase189_audit_events.sql`  
3. New `src/services/audit_writer.py` — `write_audit_event(...)` — best-effort, non-blocking (like `dead_letter.py`)  
4. Inject into the 3 mutation endpoints  
5. New `src/api/audit_router.py` — `GET /admin/audit?booking_id=&actor_id=&limit=` — JWT + admin role  
6. Contract tests (Groups A-C): write path, read path, auth
