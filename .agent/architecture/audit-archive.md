# Audit & Archive System
## iHouse Core — Canonical Platform Behavior

> Phase 887d: Expanded from stub to canonical definition.

---

## Core Principle

Nothing disappears.
Everything is traceable.
Archives are permanent. Pruning requires proof.

---

## 1. Audit Log Requirements

Every business event must record:

- Timestamp (`occurred_at`)
- Entity (`property_id`, `booking_id`, `task_id` as applicable)
- Action / event kind (`BOOKING_CREATED`, `BOOKING_AMENDED`, `BOOKING_CANCELED`, etc.)
- Previous state → New state (via `payload_json`)
- Triggering actor (system, admin, worker, OTA source)
- Notes (optional)

Audit event rows are **immutable**.
No application-level deletion is permitted on `BOOKING_CREATED` or `BOOKING_CANCELED` rows — ever.
Redaction of sensitive media (passport photos) is permitted, but the event record itself remains.

---

## 2. Automatic Semi-Annual Audit Archive (Platform Default)

**This is a platform behavior, not a feature.**
Every iHouse Core tenant receives formal audit archives automatically.
No admin action is required to activate it. It cannot be disabled.

### Cadence (anchored to tenant activation date)

| Milestone | Action | Pruning? |
|-----------|--------|---------|
| T + 6 months | Generate 6-month snapshot archive | No — live rows retained |
| T + 12 months | Generate annual archive | Yes — prune eligible rows after confirmation |
| T + 18 months | Generate 6-month snapshot archive | No |
| T + 24 months | Generate annual archive | Yes — prune eligible rows after confirmation |
| Repeat every 6 months | Annual prune at 12-month marks only | — |

### What the system does at each cycle

1. Collect all archive-eligible `event_log` rows for the tenant for the covered period
2. Generate a structured JSON export
3. Generate a PDF audit summary document
4. Bundle both into a `.zip` archive package
5. Upload to Supabase Storage (`event-archives` bucket, tenant-scoped path)
6. Insert a metadata row into `event_archives` table
7. Mark status as `pending_confirmation`
8. Notify tenant admin in-product: "Your audit archive for [period] is ready. Please review and confirm."

### Archive content tiers

**Tier 1 — Always included, never pruned:**
- `BOOKING_CREATED` events
- `BOOKING_CANCELED` events
- Manual admin property and booking actions

**Tier 2 — Included in archive, eligible for live pruning after 12 months:**
- `BOOKING_AMENDED` events (real field changes only — post-Phase 887d fix)
- `STATE_UPSERT`, `envelope_received` and other system events

**Tier 3 — Excluded from archive (pure noise, already identified):**
- `BOOKING_AMENDED` rows generated by the pre-fix iCal loop (no real field change, `source = 'ical'`)

### Archive package format

Each archive contains three files bundled as `.zip`:

| File | Format | Purpose |
|------|--------|---------|
| `audit_summary.pdf` | PDF | Human-readable. Cover page, per-property event timeline, checksum, generation timestamp. For compliance review, owner statements, legal discovery. |
| `events_export.json` | JSON | Machine-readable. Full raw export of all included rows, grouped by `property_id → booking_id → events`. Includes manifest header. |
| `manifest.json` | JSON | Archive identity: `tenant_id`, `period_start`, `period_end`, `generated_at`, `row_counts_by_kind`, `sha256_checksum`. |

## 3. Archive Storage and Access

**Storage location:** Supabase Storage — `event-archives` bucket
**Path structure:** `/{tenant_id}/{archive_id}/audit_archive_{period}.zip`
**Bucket policy:** Service-role write only. No public access. Signed URLs for authorized downloads.

**Product UI location:** Admin → Compliance → Audit Archives

Each archive is surfaced as a permanent card in the Compliance section. The UI provides:
- View summary
- Download PDF
- Download JSON bundle
- Status (pending confirmation / confirmed / pruned)

### Access rules

| Action | Permission |
|--------|-----------|
| View archive card | Tenant admin only |
| Download archive | Tenant admin only |
| Delete archive | ❌ Not available to any role via UI |
| Modify archive | ❌ Not available — archives are write-once |
| Admin cross-tenant view | Not available (future: super-admin capability) |

**Archives are immutable and permanent.**
An admin who can delete their own audit archive defeats the purpose of the system.
This constraint exists to protect operators, property owners, and the platform — not to restrict access.

---

## 4. Manual On-Demand Audit

In addition to the automatic semi-annual cycle, tenant admins may run a manual audit at any time. This produces the same format package but is:
- Triggered by admin, not the system scheduler
- Labeled as `source: manual` in the archive metadata
- Not tied to the pruning cycle
- Stored permanently alongside automatic archives

The two tracks (automatic vs manual) are distinct capabilities. Manual archives do not replace automatic archives and do not affect the pruning schedule.

---

## 5. Pruning — Safety Rules

**Pruning only occurs at the 12-month annual archive cycle, never at 6-month snapshots.**

### What can be pruned (after annual archive + confirmation)
- `BOOKING_AMENDED` rows older than 12 months
- `STATE_UPSERT`, `envelope_received` rows older than 6 months

### What is never pruned
- `BOOKING_CREATED` rows — permanent
- `BOOKING_CANCELED` rows — permanent

### Required safety checks before any `DELETE` executes

All five must pass or pruning is aborted:

1. **Storage object exists** — verify via Storage API `stat()`, not just `event_archives` metadata row
2. **Row count verified** — re-count `event_log` rows for period; must match `manifest.row_count` exactly
3. **Checksum verified** — re-compute SHA-256 of JSON export; must match `manifest.sha256_checksum`
4. **Admin confirmed** — admin clicked "Confirm Archive & Authorize Pruning"; action logged with user ID and timestamp
5. **Confirmation not expired** — admin confirmation window is 30 days; expired confirmations require re-confirmation

### Failure handling

| Failure scenario | System response |
|-----------------|-----------------|
| Archive generation fails | Retry 3× over 24 hours. If all fail: notify admin, pause cycle, **do not prune**. |
| Storage upload fails | Treat as generation failure. **Do not prune.** |
| Storage succeeds, pruning fails | Archive is safe. Mark `prune_failed`. Admin can retry manually. Live rows remain. |
| Admin does not confirm within 30 days | Re-notify. Do not prune. Archive remains in `pending_confirmation`. |

**The system never prunes live event_log rows automatically.**
Pruning always requires explicit admin confirmation of a verified archive.

## 6. `event_archives` Table Schema

```sql
CREATE TABLE event_archives (
    archive_id        TEXT PRIMARY KEY,          -- e.g. "arch-{tenant}-{yyyymm}"
    tenant_id         TEXT NOT NULL,
    period_start      TIMESTAMPTZ NOT NULL,
    period_end        TIMESTAMPTZ NOT NULL,
    source            TEXT NOT NULL,             -- 'automatic' | 'manual'
    storage_path      TEXT NOT NULL,             -- full Supabase Storage path
    storage_checksum  TEXT,                      -- SHA-256 of JSON export
    row_count         INTEGER NOT NULL,
    row_counts_by_kind JSONB,                    -- { "BOOKING_CREATED": 23, ... }
    status            TEXT NOT NULL DEFAULT 'pending_confirmation',
                                                 -- pending_confirmation | confirmed | prune_failed | pruned
    created_at        TIMESTAMPTZ DEFAULT NOW(),
    confirmed_at      TIMESTAMPTZ,
    confirmed_by      TEXT,                      -- user_id of confirming admin
    pruned_at         TIMESTAMPTZ,
    pruned_row_count  INTEGER
);
```

## 7. Invariants (Blast Constitution additions)

These join the system's inviolable rules:

> **INV-ARCHIVE-01:** The system must never execute `DELETE` on `event_log` rows unless all five pruning safety checks pass and an `event_archives` row with `status = 'confirmed'` exists for the covered period.

> **INV-ARCHIVE-02:** `BOOKING_CREATED` and `BOOKING_CANCELED` rows in `event_log` are permanent records. They must never be pruned under any condition.

> **INV-ARCHIVE-03:** Archive packages stored in Supabase Storage are write-once. No application code path may overwrite or delete them.

> **INV-ARCHIVE-04:** Archive generation failure must never trigger pruning. The live table is always the safe state when an archive is absent or unverified.

## 8. Booking Change Logging

When a booking is:
- Created → write `BOOKING_CREATED`
- Modified (real field change: `check_in`, `check_out`, `guest_name`, `status`) → write `BOOKING_AMENDED`
- Cancelled → write `BOOKING_CANCELED`
- Date changed externally via iCal → write `BOOKING_AMENDED` **only if a field actually changed**

The Phase 887d fix (2026-03-25) eliminated unconditional `BOOKING_AMENDED` writes on iCal re-sync. Only genuine field changes now produce amendment events.

---

## 9. Property Lifecycle Logging

When a property is:
- Archived → log status change
- Unarchived → log status change
- Permanently deleted (rejected/archived only) → log deletion with actor and timestamp before delete executes

The permanent delete capability (Phase 887d) hard-deletes the property and all associated records. The deletion action itself is logged in an audit entry before execution.
